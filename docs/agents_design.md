# 智能体设计文档

## 概述

本系统设计了 **5个专业化运维智能体**，每个智能体具有明确的角色定位、专业能力和工具配置。智能体通过 crewAI 框架协同工作，共同完成复杂的 DevOps 运维任务。

---

## 1. 基础设施监控专家（Infrastructure Monitor）

### 1.1 角色定义

| 属性 | 值 |
|------|---|
| 角色标识 | `infrastructure_monitor` |
| 角色名称 | 基础设施监控专家 |
| 职责定位 | 系统的"眼睛"，持续感知基础设施状态 |
| 协作权限 | 不允许委托（自主完成监控任务） |

### 1.2 角色背景

拥有10年经验的基础设施监控专家，熟练掌握 Prometheus、Grafana 等监控工具，能从海量监控数据中快速识别异常模式。座右铭："发现问题要早于用户"。

### 1.3 配置的工具

| 工具 | 功能 |
|------|------|
| `get_cpu_usage` | 获取各节点 CPU 使用率和负载 |
| `get_memory_usage` | 获取内存使用率和可用空间 |
| `get_disk_usage` | 获取磁盘使用率和 IO 情况 |
| `check_service_health` | 检查所有服务的健康状态 |
| `get_alerts` | 获取当前活跃告警列表 |
| `list_pods` | 查看 K8s Pod 运行状态 |
| `list_containers` | 查看 Docker 容器状态 |

### 1.4 主要任务

1. **基础设施监控**（`monitor_infrastructure_task`）
   - 全面扫描系统资源使用情况
   - 识别超阈值的异常指标
   - 评估各服务健康状态
   - 产出：监控报告（含严重程度分级）

2. **告警检查**（`check_alerts_task`）
   - 收集并汇总当前所有告警
   - 按严重程度分类排序
   - 初步分析告警原因
   - 产出：告警处理建议

### 1.5 输出示例

```
## 监控报告 - 2026-04-13 07:20

### 异常指标
| 指标 | 当前值 | 阈值 | 严重程度 |
|------|--------|------|---------|
| node-3 CPU | 88.9% | 85% | 🔴 紧急 |
| node-3 内存 | 88.8% | 85% | 🔴 紧急 |
| worker-service 状态 | DOWN | - | 🔴 紧急 |
| API 响应时间 | 2340ms | 2000ms | 🟡 警告 |

### 建议
立即检查 node-3 资源过载原因，优先处理 worker-service 宕机问题。
```

## 2. 故障诊断专家（Fault Diagnostician）

### 2.1 角色定义

| 属性 | 值 |
|------|---|
| 角色标识 | `fault_diagnostician` |
| 角色名称 | 故障诊断专家 |
| 职责定位 | 系统的"大脑"，分析问题根因 |
| 协作权限 | 不允许委托（独立完成诊断分析） |

### 2.2 角色背景

资深故障诊断专家，拥有丰富的大规模分布式系统故障处理经验。善于从纷繁复杂的日志和指标中抽丝剥茧，找出真正的问题根源。

### 2.3 配置的工具

| 工具 | 功能 |
|------|------|
| `search_logs` | 按关键词搜索相关日志 |
| `analyze_errors` | 深度分析错误模式和根因 |
| `get_log_statistics` | 获取日志统计和健康评分 |
| `filter_by_level` | 按级别过滤关键日志 |
| `get_container_logs` | 获取容器详细日志 |
| `get_pod_logs` | 获取 K8s Pod 日志 |
| `get_pod_status` | 获取 Pod 详细状态 |
| `get_cpu_usage` | 关联分析 CPU 指标 |
| `get_memory_usage` | 关联分析内存指标 |

### 2.4 主要任务

1. **故障诊断**（`diagnose_fault_task`）
   - 搜索和分析相关错误日志
   - 建立故障假设并用数据验证
   - 关联分析多源信息
   - 产出：诊断报告（含根因、置信度）

2. **性能分析**（`analyze_performance_task`）
   - 分析性能瓶颈指标
   - 识别慢查询和资源热点
   - 产出：性能瓶颈分析报告

### 2.5 诊断方法论

```
1. 初步假设：根据故障现象提出候选根因
2. 数据收集：通过工具收集日志、指标等证据
3. 假设验证：用数据验证每个假设
4. 根因确认：确定最可能的根因（含置信度）
5. 因果链：描述完整的故障传播路径
```

## 3. 自动修复专家（Auto Remediator）

### 3.1 角色定义

| 属性 | 值 |
|------|---|
| 角色标识 | `auto_remediator` |
| 角色名称 | 自动化修复专家 |
| 职责定位 | 系统的"手"，执行修复动作 |
| 协作权限 | 不允许委托（直接执行修复） |

### 3.2 角色背景

自动化运维专家，相信"自动化是运维的未来"。每次操作前都会制定详细的回滚计划，深知生产环境操作的风险。

### 3.3 配置的工具

| 工具 | 功能 |
|------|------|
| `restart_container` | 重启 Docker 容器 |
| `restart_pod` | 重启 K8s Pod |
| `scale_deployment` | 扩缩容 Deployment |
| `check_service_health` | 验证修复效果 |
| `get_alerts` | 确认告警是否消除 |
| `get_pod_status` | 验证 Pod 状态 |
| `list_pods` | 查看修复后 Pod 列表 |

### 3.4 主要任务

1. **执行修复**（`execute_remediation_task`）
   - 评估修复方案的风险
   - 执行立即缓解措施
   - 验证修复效果
   - 产出：修复执行报告（`remediation_report.md`）

2. **服务重启**（`restart_services_task`）
   - 安全重启指定服务
   - 确认重启成功
   - 产出：重启报告

### 3.5 修复原则

```
1. 安全第一：操作前制定回滚计划
2. 最小影响：优先选择影响范围最小的操作
3. 分步执行：先执行缓解措施，再做根本修复
4. 验证闭环：每步操作后验证效果
5. 文档记录：完整记录操作日志
```

## 4. 部署运维专家（Deployment Engineer）

### 4.1 角色定义

| 属性 | 值 |
|------|---|
| 角色标识 | `deployment_engineer` |
| 角色名称 | 部署运维专家 |
| 职责定位 | 应用发布的"保障者" |
| 协作权限 | 允许委托（可协调其他智能体协助） |

### 4.2 角色背景

DevOps 工程师，在 CI/CD 领域有深厚积累。推崇"基础设施即代码"的理念，致力于让部署过程可重复、可审计、可回滚。

### 4.3 配置的工具

| 工具 | 功能 |
|------|------|
| `get_latest_commits` | 获取最新代码提交 |
| `create_branch` | 创建发布分支 |
| `merge_branch` | 合并代码分支 |
| `get_commit_history` | 查看变更历史 |
| `get_diff` | 查看具体代码差异 |
| `build_image` | 构建 Docker 镜像 |
| `list_containers` | 查看容器状态 |
| `scale_deployment` | K8s 部署操作 |
| `list_pods` | 查看 Pod 状态 |
| `get_pod_status` | 验证部署结果 |

### 4.4 主要任务

1. **应用部署**（`deploy_application_task`）
   - 检查代码变更内容
   - 构建和推送 Docker 镜像
   - 执行 K8s 滚动更新
   - 验证部署成功
   - 产出：部署报告（`deployment_report.md`）

2. **回滚操作**（`rollback_deployment_task`）
   - 确认回滚目标版本
   - 执行快速回滚
   - 验证服务恢复
   - 产出：回滚报告

### 4.5 部署工作流

```
代码检查 → 镜像构建 → 推送镜像仓库
    ↓
K8s 滚动更新（逐个 Pod 替换）
    ↓
就绪探针检查（readinessProbe）
    ↓
流量切换
    ↓
健康验证 → 完成 / 失败自动回滚
```

## 5. 性能优化专家（Performance Optimizer）

### 5.1 角色定义

| 属性 | 值 |
|------|---|
| 角色标识 | `performance_optimizer` |
| 角色名称 | 性能优化专家 |
| 职责定位 | 系统的"调优师" |
| 协作权限 | 不允许委托（独立完成分析） |

### 5.2 角色背景

性能工程师，擅长使用火焰图、链路追踪、压力测试等工具定位性能瓶颈。优化方案往往能在不增加硬件成本的情况下，将系统吞吐量提升数倍。

### 5.3 配置的工具

| 工具 | 功能 |
|------|------|
| `get_cpu_usage` | 分析 CPU 使用模式 |
| `get_memory_usage` | 分析内存使用趋势 |
| `get_disk_usage` | 分析磁盘 IO 瓶颈 |
| `get_log_statistics` | 获取日志健康评分 |
| `analyze_errors` | 分析性能相关错误 |
| `check_service_health` | 评估服务响应性能 |
| `list_pods` | 分析 Pod 资源利用率 |
| `get_pod_status` | 获取 Pod 详细性能数据 |

### 5.4 主要任务

1. **性能优化**（`optimize_performance_task`）
   - 分析多维度性能指标
   - 识别优化机会
   - 制定优先级排序的优化方案
   - 量化预期收益
   - 产出：优化方案报告（`optimization_report.md`）

2. **优化报告**（`generate_optimization_report_task`）
   - 生成周期性性能报告
   - 与历史基线对比
   - 产出：执行级别性能报告

### 5.5 优化框架

```
1. 指标收集：全面收集 CPU/内存/IO/网络/应用层指标
2. 基线对比：与历史正常状态对比
3. 瓶颈识别：确定性能瓶颈所在层次
   - 应用层：算法复杂度、代码效率
   - 数据库层：查询优化、索引策略
   - 基础设施层：资源配置、网络带宽
4. 方案设计：按优先级（P1/P2/P3）制定优化方案
5. 效果预测：量化每项优化的预期收益（延迟/吞吐量/成本）
6. 风险评估：评估每项优化的实施风险
```

## 6. 智能体协同机制

### 6.1 信息传递

智能体之间通过任务上下文（Task Context）传递信息：

```python
# crewAI 顺序执行时，前一任务的输出自动作为后续任务的上下文
Crew(
    agents=[monitor_agent, diagnosis_agent],
    tasks=[monitor_task, diagnosis_task],  # monitor_task 结果注入 diagnosis_task
    process=Process.sequential,
)
```

### 6.2 工具共享

部分工具被多个智能体共享：

- `get_cpu_usage`：监控专家、优化专家共用
- `get_memory_usage`：监控专家、诊断专家、优化专家共用
- `list_pods`：监控专家、修复专家、部署专家共用

### 6.3 输出规范

每个智能体的输出遵循统一格式：

```markdown
## [智能体角色] 执行报告

### 摘要
执行结果的简要描述

### 详细分析/操作
具体分析内容或执行步骤

### 结论/建议
明确的结论和下一步建议
```
